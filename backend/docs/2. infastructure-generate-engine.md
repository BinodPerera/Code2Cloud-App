# Code2Cloud: Core Infrastructure & Generation Engine

This directory contains the core backend services for **Code2Cloud**—the repository analyzer and automated deployment engine. The engine is built using **Python FastAPI** and utilizes a hybrid template-driven model to generate clean, scalable, and secure production-grade cloud configurations.

---

## 🚀 Architectural Overview

To ensure industry best practices and absolute scalability, the system avoids generating monolithic scripts. Instead, it employs a **domain-driven separation of concerns** using a rule-based template injection pipeline.

### Rendering Engine Pipeline
1. **Repository Ingestion & Monorepo Detection:** React frontend authenticates the user via GitHub and transmits analyzed tech-stack metadata. The FastAPI engine detects whether the repository is a standard single-project or a multi-component monorepo (e.g. React frontend + Node backend).
2. **Dynamic Template Routing:** FastAPI routes identified sub-project tech stacks to isolated **Jinja2 templates** (e.g. NodeJS/Express, Python/FastAPI, Java/SpringBoot, and orchestrators like Docker Compose).
3. **Environment & Variable Injection:** Component-specific properties (ports, environment vars, DB connection bounds) are injected. For monorepos, connection variables are auto-wired (e.g., dynamically injecting backend URLs into the frontend setup).
4. **Hybrid Storage Persistence:**
   * **Hot Tier (Database):** Generated codes are mapped as key-value pairs (using relative workspace file paths, supporting folder levels) and saved directly into the **Central Database Ledger** to allow real-time UI previews and live Monaco Editor updates.
   * **Cold Tier (Object Storage):** Concurrently, files are zipped in-memory (`io.BytesIO()`) inside RAM and uploaded to an **AWS S3** bucket for optimized, single-transaction history lookup and bulk zip downloads.

## 💾 Storage & Ledger Architecture

To maintain high throughput, rapid UI rendering, and systemic scalability, Code2Cloud decouples the generation outputs from localized compute storage. The platform utilizes a specialized **Hybrid Storage Pattern** dividing tasks between a rapid document database state and structured cloud object storage.

### 🔄 The Dual-Tier Pipeline Diagram

1. **Database Layer (Hot Tier / Live View & Edit):** Raw generated scripts are mapped into text properties inside the database tracking ledger. This allows the frontend Monaco Editor to instantly retrieve and modify code blocks without performing expensive object store unzipping loops.
2. **In-Memory Streaming:** When bundling for cold storage, files are compressed into a single zip structure using an in-memory byte buffer (`io.BytesIO()`) inside RAM.
3. **Object Storage Layer (Cold Tier / Bulk Download):** The binary zip stream is uploaded directly to AWS S3. This reduces S3 API call overheads (1 PUT operation vs multiple individual file puts) and minimizes operational cloud costs.
